❌ Errors You Were Getting (and Why)

ERROR 1 — Seen in Node.js terminal
❌ Failed to store refresh token: new row violates row-level security policy for table "refresh_tokens"
✅ ROOT CAUSE & FIX

The refresh_tokens table had RLS (Row Level Security) enabled but no INSERT policy for the service_role. Even though your backend uses the Supabase service key, PostgREST still checks RLS policies unless an explicit bypass exists.

Fix: Run backend/database/rls-fix-v5.1.sql in the Supabase SQL Editor. This adds FOR ALL TO service_role USING (true) policies to all tables.

ERROR 2 — Seen in browser console / app UI
⚠️ Session expired. Please log in again.
✅ ROOT CAUSE & FIX (6 root causes)
#Root CauseFix Applied
1 Tokens stored in sessionStorage — cleared on tab close or page refresh PersistentTokenStore now uses localStorage in js/auth-persistent.js
2 No proactive refresh — waited until 401 error before attempting refresh SmartRefresh.scheduleRefresh() proactively refreshes at 80% of TTL
3 Refresh endpoint called generateLink (magic-link) which returns a URL, not a JWT auth.js v5.1 uses admin.createSession() → falls back to jwt.sign() with JWT_SECRET
4 After 401, the API client immediately showed "Session expired" without retrying api-client.js retries once after 401 by calling refreshAccessToken()
5 Multiple concurrent refresh calls racing each other causing token invalidation SmartRefresh uses a lock (_refreshLock) to deduplicate concurrent calls
6 RLS policy on refresh_tokens blocked all backend INSERT operations rls-fix-v5.1.sql adds service_role full-access policies
ERROR 3 — When running seed script
❌ SEED ERROR: Tenant or user not found
✅ ROOT CAUSE & FIX

The old seed script used require('pg') (raw PostgreSQL driver) and referenced DATABASE_URL which doesn't exist in this project. Column names were also wrong (slug instead of short_name, is_active instead of status).

Fix: seed-mahmoud.js has been completely rewritten to use the Supabase JS SDK with correct column names.

📌 Auth Token Flow (v5.1)
  1. User logs in → POST /api/auth/login → Supabase validates credentials
  2. Backend returns: access_token (15 min) + refresh_token (30 days)
  3. Frontend stores both in localStorage via PersistentTokenStore
  4. At 80% of TTL (≈12 min), SmartRefresh automatically calls POST /api/auth/refresh
  5. Refresh rotates tokens: old refresh token revoked, new tokens issued
  6. On page reload: SessionRestoreGate reads localStorage and silently refreshes if needed
  7. On tab close + reopen: session is fully restored from localStorage

Step 2 — Run RLS Fix SQL (CRITICAL)

2

Fix the Row Level Security policies FIXES THE ERROR

Error this fixes:
❌ Failed to store refresh token: new row violates row-level security policy for table "refresh_tokens"

Instructions:

  1. Open Supabase Dashboard → SQL Editor
  2. Click "New Query"
  3. Copy the SQL below → Paste → Click Run

Or run the full file: backend/database/rls-fix-v5.1.sql

-- ══ MINIMAL RLS FIX (run this in Supabase SQL Editor) ══

-- 1. Enable RLS on refresh_tokens
ALTER TABLE public.refresh_tokens ENABLE ROW LEVEL SECURITY;

-- 2. Drop conflicting policies
DROP POLICY IF EXISTS "service_role_full_access_rt" ON public.refresh_tokens;
DROP POLICY IF EXISTS "user_read_own_tokens"        ON public.refresh_tokens;
DROP POLICY IF EXISTS "user_update_own_tokens"      ON public.refresh_tokens;
DROP POLICY IF EXISTS "postgres_full_access_rt"     ON public.refresh_tokens;

-- 3. Service role gets FULL access (fixes the INSERT error)
CREATE POLICY "service_role_full_access_rt"
  ON public.refresh_tokens
  FOR ALL
  TO service_role
  USING (true)
  WITH CHECK (true);

-- 4. Postgres superuser also gets full access
CREATE POLICY "postgres_full_access_rt"
  ON public.refresh_tokens
  FOR ALL
  TO postgres
  USING (true)
  WITH CHECK (true);

-- 5. Enable RLS on login_activity
ALTER TABLE public.login_activity ENABLE ROW LEVEL SECURITY;
DROP POLICY IF EXISTS "service_role_full_access_la" ON public.login_activity;
CREATE POLICY "service_role_full_access_la"
  ON public.login_activity
  FOR ALL
  TO service_role
  USING (true)
  WITH CHECK (true);

-- 6. Grant ALL to service_role on all public tables
GRANT ALL ON ALL TABLES    IN SCHEMA public TO postgres, service_role;
GRANT ALL ON ALL SEQUENCES IN SCHEMA public TO postgres, service_role;
GRANT USAGE ON SCHEMA public TO postgres, service_role, authenticated, anon;

-- Verify it worked
SELECT tablename, policyname, roles, cmd
FROM pg_policies
WHERE schemaname = 'public' AND tablename IN ('refresh_tokens','login_activity')
ORDER BY tablename, policyname;
✅ Expected output in SQL Editor

You should see the policies table showing service_role_full_access_rt for both tables with cmd = ALL.

The NOTICE at the bottom should say: "RLS Fix v5.1 Applied Successfully"

Why the service_role key still hits RLS:

When you use SUPABASE_SERVICE_KEY in your Node.js backend, PostgREST connects as the service_role PostgreSQL role. In Supabase's hosted PostgREST, the service_role bypasses RLS automatically at the PostgREST level. However, if a table has RLS enabled and the explicit SECURITY DEFINER attribute is missing or the role mapping is wrong, some PostgreSQL operations may still be blocked. The safest approach is to always add explicit policies for service_role as done here.

const https = require('https');

const BACKEND = 'https://wadjet-eye-ai.onrender.com';
// For local: const BACKEND = 'http://localhost:4000';

function post(url, data) {
  return new Promise((resolve, reject) => {
    const body = JSON.stringify(data);
    const u = new URL(url);
    const opts = {
      hostname: u.hostname,
      port:     u.port || (u.protocol === 'https:' ? 443 : 80),
      path:     u.pathname,
      method:   'POST',
      headers:  { 'Content-Type': 'application/json', 'Content-Length': Buffer.byteLength(body) },
    };
    const req = (u.protocol === 'https:' ? https : require('http')).request(opts, res => {
      let d = '';
      res.on('data', c => d += c);
      res.on('end', () => {
        try { resolve({ status: res.statusCode, body: JSON.parse(d) }); }
        catch(e) { resolve({ status: res.statusCode, body: d }); }
      });
    });
    req.on('error', reject);
    req.write(body);
    req.end();
  });
}

    body = json.dumps(data).encode()

        url,
        data=body,
        headers={"Content-Type": "application/json"},
        method="POST"

# test-auth.ps1 — Full session test
# Usage: .\test-auth.ps1
# Or with custom backend: .\test-auth.ps1 -BackendUrl "http://localhost:4000"

param(
    [string]$BackendUrl = "https://wadjet-eye-ai.onrender.com",
    [string]$Email      = "mahmoud@mssp.com",
    [string]$Password   = "Admin@2024Wadjet!"
)

function Invoke-Api($method, $path, $body = $null, $token = $null) {
    $headers = @{ "Content-Type" = "application/json" }
    if ($token) { $headers["Authorization"] = "Bearer $token" }
    $params = @{
        Uri     = "$BackendUrl$path"
        Method  = $method
        Headers = $headers
    }
    if ($body) { $params["Body"] = ($body | ConvertTo-Json) }
    try {
        return Invoke-RestMethod @params
    } catch {
        $msg = $_.ErrorDetails.Message
        try { $err = $msg | ConvertFrom-Json; Write-Host "  API Error: $($err.error)" -ForegroundColor Red }
        catch { Write-Host "  HTTP Error: $($_.Exception.Message)" -ForegroundColor Red }
        return $null
    }
}
