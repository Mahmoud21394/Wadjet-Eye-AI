1. Root Cause Analysis — Why Sessions Expired Immediately

TL;DR: The access token was stored in sessionStorage, which is wiped on every page refresh or tab close. When the page reloaded, the token was gone, and the refresh token in localStorage was never used to get a new one in time — so the app treated it as a new unauthenticated session.

There were 6 compounding bugs. They are listed from most to least critical:

#BugImpactLocation
BUG 1 Access token in sessionStorage — cleared on every tab close / page refresh 🔴 Critical — user logged out on every F5 js/api-client.js — TokenStore.set()
BUG 2 Session restore fired before DOMContentLoaded — UI transition (hide login, show dashboard) never triggered 🔴 Critical — even when token was valid, user saw login screen js/api-client.js — _autoRestoreSession()
BUG 3 Render.com cold-start timeout (≈15 s) — refresh fetch timed out after 5 s, returned false, cleared all tokens 🔴 Critical — session destroyed by timeout even though token was valid js/api-client.js — _doRefresh() no retry logic
BUG 4 auth:expired fired on first 401 with no retry — immediately called doLogout() 🟠 High — one transient HTTP error logged user out js/api-client.js — initRealtime()
BUG 5 Emergency/offline token stored in sessionStorage — also wiped on refresh, so offline login didn't persist either 🟠 High — "bypass-demo-token" sessions ended on every reload js/main.js — _doEmergencyLogin()
BUG 6 No proactive refresh timer — refresh only happened reactively (on 401), but by then the 401 itself had already triggered logout 🔵 Medium — race condition between expiry and logout Missing feature — nothing refreshed the token before it expired

2. How Supabase JWT Works

Token Lifecycle

User logs in
→
Backend issues Access Token (15 min TTL)
+
Refresh Token (30 days TTL)
→
Frontend stores both
Access token expires
→
POST /api/auth/refresh with refresh_token
→
New Access Token issued
→
Old refresh token rotated (new one issued)
Key Supabase behaviour: Refresh tokens are rotated on every use. This means once you call /refresh, the old refresh token is invalidated. If two tabs call it simultaneously, one will get a 401. The fix uses a single-flight guard to prevent this.

Why Render.com Makes This Worse

Render's free/hobby tier spins down after 15 minutes of inactivity. Cold start takes 10–20 seconds. If the refresh fetch times out in under 12 s, the backend hasn't woken up yet — the refresh "fails" but the token is actually still valid.

The old code had a 5-second timeout with no retry. The fix uses a 12-second timeout with 4 retry attempts and exponential backoff (2s, 4s, 8s).

6. How It Works Now — Full Flow

A) First Login

1

User submits credentials

main.js doLogin() calls API.auth.login(email, password, tenant).

2

Backend returns JWT

Response: { token, refreshToken, expiresAt, user }

3

Tokens persisted to localStorage

PersistentAuth_onLogin() is called, storing access + refresh tokens in localStorage under keys we_v5_access, we_v5_refresh, we_v5_expiry, we_v5_user.

4

SessionWatchdog starts

Checks every 4 minutes. If token expires in < 5 minutes, silently refreshes.

B) Page Refresh / New Tab

1

auth-persistent.js loads

Before DOMContentLoaded: patches TokenStore, refreshAccessToken, and auth:expired handler.

2

DOMContentLoaded fires

PersistentAuth_restoreOnLoad() runs:
1. Migrates any old tokens from legacy keys
2. Calls SessionRestoreGate.restore()

3

Session restored from localStorage

If we_v5_access is valid → builds CURRENT_USER → fires auth:restored event.

If expired but refresh token exists → calls SmartRefresh.refresh() first.

4

UI transitions to dashboard

The auth:restored listener in index.html hides the login screen and shows the dashboard with a "Welcome back" toast.

C) Token Expires While User is Active

1

Watchdog detects expiry approaching

Every 4 minutes, checks if token expires within 5 minutes. If yes → calls SmartRefresh.refresh() immediately.

2

SmartRefresh POSTs to /api/auth/refresh

Up to 4 attempts, 12-second timeout per attempt, exponential backoff (2s, 4s, 8s). This handles Render cold starts transparently.

3

New tokens stored, watchdog continues

New access + refresh tokens saved to localStorage. User stays logged in with zero interruption.

7. Emergency / Offline Sessions

When the Supabase backend returns User profile not found or is unreachable, the app falls back to a hardcoded _EMERGENCY_ACCOUNTS list for known admins:

EmailRoleCondition
mahmoud.osman@wadjet.aisuper_adminAny password ≥ 6 chars
mahmoud@mssp.comsuper_adminAny password ≥ 6 chars
admin@mssp.comadminAny password ≥ 6 chars
analyst@mssp.comanalystAny password ≥ 6 chars
Before fix: Offline sessions used a bypass-demo-token-xxx stored only in sessionStorage, so they vanished on page refresh.

After fix: PersistentTokenStore.setOffline(token) stores the token in localStorage with a 30-day TTL and sets the offline flag. On page reload, SessionRestoreGate detects the offline flag and restores the session without calling the backend.

9. Backend Requirements

The frontend fix alone handles 90% of session issues. But the backend must also be configured correctly for real Supabase auth (not offline mode).

Supabase Environment Variables

# backend/.env

SUPABASE_ANON_KEY=eyJ...
SUPABASE_SERVICE_ROLE_KEY=eyJ...    # Used to bypass RLS for profile lookups
JWT_SECRET=your-supabase-jwt-secret # Must match Supabase project JWT secret

REFRESH_TOKEN_TTL=2592000           # 30 days (seconds)
Critical: The JWT_SECRET in your backend .env MUST match the JWT secret in your Supabase project settings (Settings → API → JWT Settings → JWT Secret). A mismatch causes all tokens to be rejected with 401 immediately.

10. Next Steps for Production

1

Seed the Supabase users table

Run the SQL above to insert the real profile row. Once done, the backend will return a proper JWT and offline mode won't be needed.

Or run: cd backend && node scripts/seed-mahmoud.js

2

Verify JWT_SECRET matches

Go to Supabase Dashboard → Settings → API → JWT Settings. Copy the JWT Secret and set it in backend/.env as JWT_SECRET.

3

Deploy updated frontend to Vercel

The 3 changed files (js/auth-persistent.js, js/main.js, index.html) need to be deployed. Push to GitHub → Vercel auto-deploys.

4

Enable Render "Always On" (optional but recommended)

In the Render dashboard for your backend service, enable Health check path = /health. This prevents cold starts and makes the 4-retry backoff less necessary.

5

Consider HTTP-Only cookies for refresh tokens

Currently refresh tokens are in localStorage (accessible by JS). For maximum security, configure the backend to set the refresh token as an HTTP-Only SameSite=Strict cookie. The credentials: 'include' flag is already set in api-client.js.

6

Remove emergency accounts before production

Once Supabase auth is fully working, remove _EMERGENCY_ACCOUNTS from js/main.js or restrict it to a build flag. These hardcoded credentials are a security risk in production.

11. Troubleshooting Guide

SymptomLikely CauseFix
Still logged out on F5 Old cached version of api-client.js or index.html Hard refresh: Ctrl+Shift+R. Clear site data in DevTools.
"Session expired" after exactly 15 min Watchdog not started, or backend /refresh returns 401 Check console for [SessionWatchdog] Started. Verify refresh endpoint.
Offline session not persisting localStorage disabled (private browsing) or storage full Check localStorage.getItem('we_v5_offline') in DevTools console.
"User profile not found" still showing Emergency fallback not triggering — email not in _EMERGENCY_ACCOUNTS Add the email to the list in js/main.js, or seed the DB.
Refresh loops (constant token refresh) Clock skew — system clock is wrong, token looks expired immediately Sync system clock. Check PersistentTokenStore.getExpiry() in console.
Two tabs fighting over refresh token Both tabs attempt refresh simultaneously, one gets 401 from rotated token Use BroadcastChannel to coordinate. Already guarded by single-flight in SmartRefresh.

12. Token Storage Comparison

MethodSurvives RefreshSurvives Tab CloseXSS RiskUsed For
sessionStorage ❌ No ❌ No Medium Legacy mirror only (backward compat)
localStorage (current) ✅ Yes ✅ Yes Medium All tokens (access, refresh, user, expiry)
HTTP-Only Cookie (recommended future) ✅ Yes ✅ Yes Low (JS can't read) Refresh token only
Memory only ❌ No ❌ No Lowest Not suitable for persistent login
Security note: localStorage is readable by JavaScript and vulnerable to XSS. The current approach is acceptable for an internal SOC tool. For a public-facing SaaS product, move the refresh token to HTTP-Only cookies. The access token can stay in memory + sessionStorage since it's short-lived.
