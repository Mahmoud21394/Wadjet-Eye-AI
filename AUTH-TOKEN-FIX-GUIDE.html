Auth Token Fix — v6.1

AUTH BROKEN FIXED

Wadjet-Eye AI · Symptom: "Auth token: NOT FOUND" after login — every API call returns 401

🔬 Root Cause — 4 Compounding Bugs

🚨
Summary: login-secure-patch.js (v6.0) was designed for httpOnly cookie auth, but the backend still returns JWT in the response body. _finalizeLogin() intentionally stored zero tokens — breaking every API call silently.
#BugFileImpact
1 CRITICAL _finalizeLogin() stored NO tokens after login
v6.0 design assumed httpOnly cookie — backend still returns Bearer JWT. Token was discarded. 		js/login-secure-patch.js Every API call: 401 — token not found
2 CRITICAL global.getAuthToken = () => null
auth-secure.js overrode the getAuthToken function to always return null, breaking all auth-validator.js token reads. 		js/auth-secure.js line 329 getToken() in auth-validator.js always null
3 CRITICAL migrateFromV5() called clearLegacyKeys() on every page load
Even if login somehow saved a token, this function deleted it again on the next DOMContentLoaded. 		js/auth-secure.js Tokens wiped on every page reload
4 CRITICAL authFetch() in auth-secure.js had no Authorization header
Cookie-only mode assumed httpOnly cookies. No Bearer header injected. Backend auth middleware requires Authorization: Bearer. 		js/auth-secure.js All API calls unauthenticated
5 INFO auth-secure.js and login-secure-patch.js were NOT in index.html
The files existed but weren't loaded — so the overrides weren't even running. 		index.html Fixes not applied — old vulnerable doLogin() was running

📝 Before vs After — Key Code Changes

1. login-secure-patch.js — _finalizeLogin() (THE CRITICAL FIX)

❌ Before (v6.0) — discards token

async function _finalizeLogin(data) {
  // Store ONLY display data — NO tokens
  const displayUser = data.user;
  if (typeof window.SecureSession?.save === 'function') {
    window.SecureSession.save(displayUser);  // only saves name/email/role
  }
  // ← token (data.token) is NEVER written to localStorage
  // ← every subsequent API call: getToken() → null → 401
}

✅ After (v6.1) — saves token via 3-tier fallback

async function _finalizeLogin(data) {
  const accessToken  = data.token || data.access_token || null;
  const refreshToken = data.refreshToken || data.refresh_token || null;

  // TIER 1: UnifiedTokenStore (auth-interceptor.js) — writes to ALL known keys
  if (typeof window.UnifiedTokenStore !== 'undefined') {
    window.UnifiedTokenStore.save({ token: accessToken, refreshToken, ... });

  // TIER 2: PersistentAuth_onLogin (auth-persistent.js)
  } else if (typeof window.PersistentAuth_onLogin === 'function') {
    window.PersistentAuth_onLogin(user, accessToken, refreshToken, ...);

  // TIER 3: Direct localStorage writes (absolute fallback)
  } else if (accessToken) {
    ['wadjet_access_token','we_access_token','tp_access_token'].forEach(k => {
      localStorage.setItem(k, accessToken);
      sessionStorage.setItem(k, accessToken);
    });
  }
  // Also sync legacy TokenStore (api-client.js)
  if (accessToken && window.TokenStore) window.TokenStore.set(accessToken, ...);
}

2. auth-secure.js — getAuthToken (was always null)

❌ Before

global.getAuthToken = () => null;  // "tokens live in cookies — not readable"
// This overrode auth-interceptor.js's real getAuthToken() → broke all token reads

✅ After

global.getAuthToken = function () {
  return localStorage.getItem('wadjet_access_token')
      || localStorage.getItem('we_access_token')
      || localStorage.getItem('tp_access_token')
      || sessionStorage.getItem('wadjet_access_token')
      || null;
};

3. auth-secure.js — authFetch (no Bearer header)

❌ Before

const fetchOpts = {
  credentials: 'include',
  headers: {
    'Content-Type': 'application/json',
    // No Authorization header — cookie handles auth (but backend has NO cookie middleware!)
  },
};

✅ After

const token = _getStoredToken();  // reads wadjet_access_token from localStorage

const fetchOpts = {
  credentials: 'include',          // cookie (future httpOnly auth)
  headers: {
    'Content-Type': 'application/json',
    ...(token ? { 'Authorization': `Bearer ${token}` } : {}),  // Bearer (current)
  },
};

4. auth-secure.js — migrateFromV5 (was deleting tokens)

❌ Before

function migrateFromV5() {
  SecureSession.clearLegacyKeys();  // deleted ALL token keys on EVERY page load!
  // This ran on DOMContentLoaded — wiped any token just saved by login
}

✅ After

function migrateFromV5() {
  // REMOVED clearLegacyKeys() — was wiping valid tokens on every page reload
  // Token cleanup now only happens on explicit logout (PersistentAuth_onLogout)
  console.info('[SecureAuth v6.1] Startup: token keys preserved');
}
