Section 3 — Production Policy (tenant-scoped)

Apply this after confirming Section 2 works. Uncomment this block in the migration file.

-- Remove debug policy first
DROP POLICY IF EXISTS "debug_iocs_read_all" ON iocs;

-- 1. Service role has unrestricted access (backend uses this)
CREATE POLICY "service_role_iocs_all" ON iocs
  FOR ALL TO service_role
  USING (true) WITH CHECK (true);

-- 2. Authenticated users only see their own tenant's IOCs
CREATE POLICY "iocs_tenant_select" ON iocs
  FOR SELECT TO authenticated
  USING (
    tenant_id = (
      SELECT u.tenant_id FROM public.users u
      WHERE u.auth_id = auth.uid() LIMIT 1
    )
  );

-- 3. Same tenant restriction for writes
CREATE POLICY "iocs_tenant_insert" ON iocs
  FOR INSERT TO authenticated
  WITH CHECK (
    tenant_id = (
      SELECT u.tenant_id FROM public.users u
      WHERE u.auth_id = auth.uid() LIMIT 1
    )
  );

Section 4 — Fix IOC Type Constraint

-- Old constraint rejected 'md5', 'sha1', 'sha256', 'hostname' etc.
-- Ingestion pipeline writes these types → silent insert failures

ALTER TABLE iocs DROP CONSTRAINT IF EXISTS iocs_type_check;

ALTER TABLE iocs ADD CONSTRAINT iocs_type_check CHECK (
  type IN (
    -- Original schema types
    'ip', 'domain', 'url', 'email', 'filename', 'registry', 'mutex', 'asn', 'cve',
    -- Hash types (both old and new naming)
    'hash_md5', 'hash_sha1', 'hash_sha256', 'hash_sha512',
    'md5', 'sha1', 'sha256', 'sha512',
    -- Additional ingestion feed types
    'hostname', 'cidr', 'hash', 'certificate'
  )
);

Section 5 — Performance Indexes

-- GIN trigram index for fast ILIKE search on value column
CREATE EXTENSION IF NOT EXISTS "pg_trgm";
CREATE INDEX IF NOT EXISTS idx_iocs_value_trgm
  ON iocs USING gin(value gin_trgm_ops);

-- Composite indexes for common query patterns
CREATE INDEX IF NOT EXISTS idx_iocs_tenant_status
  ON iocs(tenant_id, status) WHERE status = 'active';

CREATE INDEX IF NOT EXISTS idx_iocs_tenant_risk
  ON iocs(tenant_id, risk_score DESC);

CREATE INDEX IF NOT EXISTS idx_iocs_tenant_type
  ON iocs(tenant_id, type);

-- Full-text search
CREATE INDEX IF NOT EXISTS idx_iocs_fts
  ON iocs USING gin(
    to_tsvector('english',
      coalesce(value,'') || ' ' || coalesce(threat_actor,'') ||
      ' ' || coalesce(malware_family,'') || ' ' || coalesce(notes,'')
    )
  );

🔑 Tenant ID Troubleshooting

Why Tenant ID Matters

IOC data was ingested to tenant 00000000-0000-0000-0000-000000000001 (the DEFAULT_TENANT). But if your user's tenant_id in the users table is different (or NULL), the query returns 0 rows even though 33,750 rows exist on tenant-001.

Run this SQL to check:

-- Check your user's tenant_id
SELECT id, email, role, tenant_id, status
FROM users
WHERE email = 'mahmoud.osman@wadjet.ai';  -- replace with your email

-- If tenant_id is NULL or different from iocs data:
UPDATE users
SET tenant_id = '00000000-0000-0000-0000-000000000001'
WHERE email = 'mahmoud.osman@wadjet.ai';

-- Verify match
SELECT
  u.email,
  u.tenant_id AS user_tenant,
  COUNT(i.id) AS ioc_count
FROM users u
LEFT JOIN iocs i ON i.tenant_id = u.tenant_id
WHERE u.email = 'mahmoud.osman@wadjet.ai'
GROUP BY u.email, u.tenant_id;

🏗️ How Supabase RLS Works (and Why It Silently Returns [])

The Silent Empty Array Trap

When Supabase RLS has no matching policy, it does not throw an error. Instead, PostgREST filters every row and returns an empty result:

  • data: [] — zero rows returned
  • count: 0 — count also shows 0
  • No error message — looks identical to "table is empty"

This is by design — RLS is a security feature and revealing "rows exist but you can't see them" would be a data leak. But it makes debugging very confusing. 

-- THE KEY INSIGHT: In SQL Editor (which uses postgres/service_role):
SELECT COUNT(*) FROM iocs;  -- Returns 33,750 ✅

-- But via PostgREST (anon key, no SELECT policy):
-- GET /rest/v1/iocs → { data: [], count: 0 }  ← SILENT EMPTY

-- SOLUTION: Add a SELECT policy OR use service_role in backend
-- The backend (Node.js) MUST use service_role key to bypass RLS

📈 Indexing Strategy for 33K+ IOCs

Index Type Purpose Query it Accelerates
idx_iocs_value_trgm GIN (pg_trgm) Text search on IOC value value ILIKE '%term%' — 33K rows in ~2ms vs 200ms
idx_iocs_tenant_status B-tree partial Active IOCs per tenant tenant_id=X AND status='active'
idx_iocs_tenant_risk B-tree Default sort by risk ORDER BY risk_score DESC
idx_iocs_tenant_type B-tree Type filter type='ip' or type='domain'
idx_iocs_fts GIN (tsvector) Full-text search Future FTS queries across multiple fields
iocs_tenant_value_uniq Unique B-tree Deduplication constraint UPSERT ON CONFLICT (tenant_id, value)

Scaling Recommendations for 100K+ IOCs

  • Partition by tenant_id — use PostgreSQL declarative partitioning when tenants have >100K rows each
  • Use HEAD requests for counts — .select('*', { count: 'exact', head: true }) avoids fetching rows just for counting
  • Cache stats — store counts in a ioc_stats_cache table, refresh every 5 minutes via cron
  • Paginate type aggregation — don't SELECT type FROM iocs LIMIT 5000 — use GROUP BY via RPC or fetch in 10K chunks
  • Async enrichment queue — don't block the insert path; use ioc_enrichment_queue table (already exists in your schema)
