🔬 Understanding the Problem READ FIRST
⚠️

What the Supabase Warning Actually Means

Supabase recently changed from a single Legacy JWT Secret (one shared HS256 symmetric key) to a new JWT Signing Keys system that supports multiple keys and asymmetric signing. The warning is informational — your old keys still work until you explicitly revoke them.

Concept Old System New System Status
JWT Signing Method HS256 (symmetric, 1 shared secret) RS256 or HS256 (per-key control) Migrating
anon key eyJhbGciOiJIUzI1NiIs... (JWT) sb_publishable_... Both valid
service_role key eyJhbGciOiJIUzI1NiIs... (JWT) sb_secret_... Both valid
JWT Secret KO3aLoS4LJqK4Nz6O+Lx... New key from Key Settings page Legacy (keep for now)
🔍 Root Cause of Your HTTP 401
❌

Error: {"error":"Missing or invalid Authorization header","code":"MISSING_TOKEN","hint":"Include: Authorization: Bearer"}

This error comes from your own backend middleware (backend/middleware/auth.js line 76-81). It means the request reached your Express server but the Authorization: Bearer <token> header was missing or empty.

The 401 chain — what's actually happening:

Frontend calls POST /api/ingest/run (or any protected route)
The JavaScript fetch does NOT include the Authorization header because it reads the token from the wrong localStorage key, or the token was never stored under the expected key.
Express middleware runs extractToken(req)
Checks for Authorization header → not found. Checks X-Access-Token header → not found. Checks query param → not found. Returns { token: null }.
401 MISSING_TOKEN is returned
The error you see: HTTP 401 — Missing or invalid Authorization header
💡

This is separate from the JWT migration warning

The JWT migration warning is about the type of API keys. Your 401 is about the frontend not sending a token at all. Both must be fixed — and this guide covers both.

🌐 Step 3 — Fix the Frontend Auth (Root Cause of 401)
🚨

This is the actual fix for your HTTP 401 error

The Supabase key update above stops the warning. This step fixes the actual MISSING_TOKEN 401 by ensuring the frontend properly stores and sends the JWT in the Authorization header.

A
Verify token storage key in js/auth-interceptor.js
The token must be stored under the correct localStorage key
▶

Your auth-interceptor.js (v5.2) defines UNIFIED_KEYS. The primary key must be what all API calls read from. Confirm this structure exists:

// In js/auth-interceptor.js — UNIFIED_KEYS must include these:
const UNIFIED_KEYS = {
  primary: {
    access:   'wadjet_access_token',    // ← PRIMARY key
    refresh:  'wadjet_refresh_token',
    expires:  'wadjet_token_expires',
    user:     'wadjet_user_data',
    session:  'wadjet_session_id',
  },
  legacy: [
    'we_access_token',       // old key
    'access_token',          // generic
    'supabase_token',        // supabase sdk
    'sb-miywxnplaltduuscjfmq-auth-token',  // supabase storage key
  ]
};

When calling any API, the request must include:

// Every authenticated API call needs this header:
const token = localStorage.getItem('wadjet_access_token')
           || localStorage.getItem('we_access_token')
           || localStorage.getItem('access_token');

fetch('/api/ingest/run', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${token}`   // ← This must be present
  }
});
B
Verify login stores token under the correct key
Check what js/main.js does on successful login
▶

After a successful POST /api/auth/login, the response contains { token, access_token, ... }. Ensure the login handler stores it correctly:

js/main.js — login handler (check this code exists)
const data = await response.json();
const accessToken = data.token || data.access_token;
 
// Store under ALL known keys for compatibility:
localStorage.setItem('wadjet_access_token', accessToken);
localStorage.setItem('we_access_token', accessToken); // legacy
localStorage.setItem('access_token', accessToken); // generic
localStorage.setItem('wadjet_refresh_token', data.refresh_token);
localStorage.setItem('wadjet_token_expires', Date.now() + (data.expires_in * 1000));
✅

Your auth-interceptor.js already handles this

The UnifiedTokenStore.save() function already writes to all keys. Make sure loginUser() calls UnifiedTokenStore.save() rather than localStorage.setItem directly.

C
Debug: Check token in browser DevTools right now
Find out why the token isn't being sent
▶

Open your app in the browser, log in, then open DevTools (F12) and run this in the Console:

// Paste this in DevTools console after logging in:
const keys = [
  'wadjet_access_token',
  'we_access_token',
  'access_token',
  'supabase_token',
  `sb-miywxnplaltduuscjfmq-auth-token`
];

console.log('=== TOKEN STORAGE DEBUG ===');
keys.forEach(k => {
  const val = localStorage.getItem(k);
  console.log(`localStorage[${k}]:`, val ? val.substring(0, 30) + '...' : 'NOT SET');
});

// Also check sessionStorage
keys.forEach(k => {
  const val = sessionStorage.getItem(k);
  if (val) console.log(`sessionStorage[${k}]:`, val.substring(0, 30) + '...');
});

Expected output: At least one key should show a token starting with eyJ... (a JWT). If ALL say NOT SET, the login is not saving the token.

Then test the API call manually:

// Test an authenticated API call from DevTools console:
const token = localStorage.getItem('wadjet_access_token')
           || localStorage.getItem('we_access_token')
           || localStorage.getItem('access_token');

if (!token) { console.error('❌ No token found — need to log in first'); }
else {
  fetch('https://wadjet-eye-ai.onrender.com/api/auth/me', {
    headers: { 'Authorization': `Bearer ${token}` }
  })
  .then(r => r.json())
  .then(d => console.log('✅ Auth working:', d))
  .catch(e => console.error('❌ Auth failed:', e));
}
