🔬 Confirmed Root Cause (Frontend Rendering Conflict)

⚠️ The API is working — this is a JavaScript function override conflict

Three JavaScript files all define window.renderIOCDatabase and they override each other in the wrong order. The version that eventually runs fetches real data but the result never updates the visible DOM elements because earlier versions already drew the container structure with different element IDs.

Script Load Order Analysis

index.html script load order (relevant files):
──────────────────────────────────────────────
 1. js/features.js         → defines renderIOCDatabase() [MOCK DATA from window._iocDB array]
 2. js/live-pages.js       → defines renderIOCDatabaseLive() + sets window.renderIOCDatabase = renderIOCDatabaseLive
 3. js/main.js             → defines PAGE_CONFIG['ioc-database'].onEnter = () => renderIOCDatabase()
                             ↑ Captures the live-pages.js version at definition time via global lookup
 4. js/ioc-intelligence.js → defines renderIOCDatabase() [REAL API] + sets window.renderIOCDatabase = renderIOCDatabase
                             ↑ Overrides window.renderIOCDatabase ← THIS IS THE CORRECT VERSION

Problem: main.js PAGE_CONFIG.onEnter calls the GLOBAL renderIOCDatabase (not window.renderIOCDatabase)
         which is the function from features.js scope if hoisting is involved.
         Live-pages.js also sets window.renderIOCDatabase but ioc-intelligence.js sets it LAST.

Result:
  - window.renderIOCDatabase = ioc-intelligence.js version (correct)
  - But PAGE_CONFIG.onEnter was registered BEFORE ioc-intelligence.js loaded
  - When user clicks IOC Database nav, it calls the cached version
  - The mock/legacy version runs, renders 10-12 fake rows from window._iocDB
  - The real API data is never displayed

Secondary Issue: IOCDB.loading Stuck State

If ioc-intelligence.js's renderIOCDatabase() throws before setting IOCDB.loading=false:
  → Every subsequent call to iocdbLoadPage() exits immediately (IOCDB.loading = true guard)
  → Table stays as "Loading…" spinner forever
  → User must hard-refresh to recover

Tertiary Issue: 12,099 vs 33,750 IOCs

12,099 = IOCs belonging to YOUR tenant (00000000-...-000000000001)
33,750 = ALL IOCs across all 3 tenants (001 + 002 + 003)

This is correct multi-tenant behavior.
SUPER_ADMIN users can now toggle "All Tenants" view via the new header button.

🔒 Production RLS Policies (Apply in Supabase)

The debug policy grants read to ALL authenticated users. Replace with these tenant-scoped production policies:

-- ── Step 1: Remove debug (broad-access) policy ──────────────────────────
DROP POLICY IF EXISTS "debug_iocs_all_authenticated" ON iocs;
DROP POLICY IF EXISTS "iocs_debug_all_read" ON iocs;

-- ── Step 2: Service role bypass (for backend service_role client) ────────
-- This is already safe: backend enforces tenant_id at application layer
DROP POLICY IF EXISTS "iocs_service_role_all" ON iocs;
CREATE POLICY "iocs_service_role_all"
ON iocs FOR ALL
TO service_role
USING (true)
WITH CHECK (true);

-- ── Step 3: Tenant-scoped SELECT for authenticated users ─────────────────
-- Users can only see IOCs that belong to their tenant
DROP POLICY IF EXISTS "iocs_tenant_read" ON iocs;
CREATE POLICY "iocs_tenant_read"
ON iocs FOR SELECT
TO authenticated
USING (
  tenant_id = (
    SELECT tenant_id FROM public.users
    WHERE auth_id = auth.uid()
    LIMIT 1
  )
);

-- ── Step 4: SUPER_ADMIN can see all tenants ──────────────────────────────
DROP POLICY IF EXISTS "iocs_super_admin_read" ON iocs;
CREATE POLICY "iocs_super_admin_read"
ON iocs FOR SELECT
TO authenticated
USING (
  EXISTS (
    SELECT 1 FROM public.users
    WHERE auth_id = auth.uid()
    AND role IN ('SUPER_ADMIN', 'ADMIN')
  )
);

-- ── Verify ───────────────────────────────────────────────────────────────
SELECT policyname, cmd, roles
FROM pg_policies
WHERE tablename = 'iocs'
ORDER BY policyname;
-- Expected: iocs_service_role_all, iocs_super_admin_read, iocs_tenant_read
💡 Why service_role bypasses RLS and still works securely

The backend uses supabaseAdmin (service_role client) which bypasses RLS entirely. Tenant isolation is enforced at the application layer with .eq('tenant_id', tenantId) in every query. This is the recommended pattern for multi-tenant backends — the DB handles performance, the app handles isolation.
