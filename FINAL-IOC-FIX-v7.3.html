🔬 Root Cause Analysis — Three Silent Crashes

After exhaustive code audit of all four JS files that define renderIOCDatabase, three independent bugs were identified. Each one alone would prevent IOCs from displaying. Together, they made the failure completely silent — no console errors, no broken UI, just an empty table.

# File Line(s) Bug Severity Effect
BUG 1 js/main.js 333, 340 ReferenceError: IOCDB is not defined
main.js references bare IOCDB variable inside PAGE_CONFIG.onEnter closure. IOCDB is declared with const in ioc-intelligence.js scope — NOT on window. Throws ReferenceError at call time, crashing the entire onEnter before renderIOCDatabase() is ever called.		 CRITICAL onEnter never fires; IOC page stays as loading spinner forever
BUG 2 js/ioc-intelligence.js 56, 68 Double /api prefix: /api/api/iocs
_apiGet called window.authFetch('/api' + finalPath). authFetch internally prepends base + '/api' to every path, creating the URL https://wadjet.../api/api/iocs which returns 404 on every API call.		 CRITICAL All API calls fail with 404; table shows error state even with valid auth
BUG 3 js/features.js 587 Function declaration collision
features.js declared function renderIOCDatabase() (mock data). JavaScript hoists function declarations to the top of the script scope. When PAGE_CONFIG.onEnter ran before ioc-intelligence.js patched it, the mock version ran instead, displaying 10 static fake IOCs.		 HIGH Mock data shown when the patch hasn't fired yet; confusing UX
Why previous fixes didn't work: All previous fix attempts (v7.0, v7.1, v7.2) targeted the RLS policy and tenant_id issues in the backend — which are real but secondary. Bug #1 caused a ReferenceError that killed the entire onEnter handler silently (JavaScript catches the error inside the try {} block in navigateTo). Bug #2 caused all API calls to return 404. Neither error appeared prominently in the browser console unless DevTools was open on the right tab at the right time.

🔧 Exact Code Changes (v7.3)

Fix 1 — js/main.js: Replace bare IOCDB with window.IOCDB

The bare IOCDB reference inside the onEnter closure threw ReferenceError because IOCDB lives in ioc-intelligence.js scope, not the global scope. Using window.IOCDB (which ioc-intelligence.js now sets) resolves this safely.

- IOCDB && (IOCDB.loading = false); // ReferenceError crash!
- // ... other code ...
- onLeave:()=>{ IOCDB && (IOCDB.loading = false); }

+ if (window.IOCDB) window.IOCDB.loading = false; // safe — set by ioc-intelligence.js
+ // ... other code ...
+ onLeave:()=>{ if (window.IOCDB) window.IOCDB.loading = false; }

Fix 2 — js/ioc-intelligence.js: Remove double /api prefix in _apiGet/_apiPost

authFetch(path) already prepends base + "/api" internally. Passing "/api/iocs" produced /api/api/iocs. The fix removes the redundant prefix from the call site.

// _apiGet — before:
- return window.authFetch(`/api${finalPath}`);
  // → authFetch receives '/api/iocs?...'
  // → builds URL: base + '/api' + '/api/iocs' = /api/api/iocs ❌ (404)

// _apiGet — after:
+ return window.authFetch(finalPath);
  // → authFetch receives '/iocs?...'
  // → builds URL: base + '/api' + '/iocs' = /api/iocs ✅

// _apiPost — same fix:
- return window.authFetch(`/api${path}`, { ... });
+ return window.authFetch(path, { ... });

Fix 3 — js/ioc-intelligence.js: Expose IOCDB as window.IOCDB

Adding window.IOCDB = IOCDB; makes the state object accessible across script boundaries, enabling main.js to reset the loading flag without a ReferenceError.

// Added in ioc-intelligence.js global exports section:
+ window.IOCDB = IOCDB;  // Expose for cross-script access (main.js, live-pages.js, etc.)

Fix 4 — js/features.js: Rename mock function to _renderIOCDatabaseMock

Renaming prevents the old mock version from being used when the ioc-intelligence.js patch hasn't fired yet. Any internal callers in features.js are updated to use the new name.

- function renderIOCDatabase() {     // hoisted to global scope, overrides production version
+ function _renderIOCDatabaseMock() { // safe internal name, no global collision

// Internal call in features.js (bulk import completion):
- renderIOCDatabase();
+ _renderIOCDatabaseMock();

📋 Script Load Order & Override Chain

Understanding which script "wins" the window.renderIOCDatabase assignment:


1. features.js    → declares function renderIOCDatabase()  ⚠ RENAMED to _renderIOCDatabaseMock in v7.3
                     → window._iocDB = [10 static mock IOCs]

2. live-pages.js  → defines renderIOCDatabaseLive() (real API)
                     → sets window.renderIOCDatabase = renderIOCDatabaseLive  (only if _iocIntelligenceLoaded is undefined)

3. main.js        → defines PAGE_CONFIG['ioc-database'].onEnter
                     → calls window.renderIOCDatabase() safely via window. reference

4. ioc-intelligence.js  → sets window.IOCDB = IOCDB                    ✅ v7.3 NEW
                          → sets window._iocIntelligenceLoaded = true
                          → sets window.renderIOCDatabase = renderIOCDatabase  ← WINS
                          → patches PAGE_CONFIG['ioc-database'].onEnter via _patchPageConfig()

5. live-pages-patch.js  → patches SOAR nav; does NOT touch ioc-database entry ✅
Key principle: Because ioc-intelligence.js is loaded as item #4 (after main.js), it both sets window.renderIOCDatabase AND patches PAGE_CONFIG['ioc-database'].onEnter via _patchPageConfig(). Any navigation to the IOC page after DOMContentLoaded will always call the production version.
