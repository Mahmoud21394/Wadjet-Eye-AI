🔍 Exact Root Cause — Why This Error Appears

The Problem in 3 Lines

1. The migration script ran ALTER TABLE refresh_tokens ENABLE ROW LEVEL SECURITY;

2. But it never created an INSERT policy — not even for service_role

3. In Supabase, when RLS is enabled with no policies → ALL rows are blocked

Common misconception: Many developers think service_role key always bypasses RLS. In standard PostgreSQL this is true, but Supabase's PostgREST layer enforces RLS even for service_role UNLESS you have an explicit policy OR the table has RLS disabled. Adding USING (true) WITH CHECK (true) for service_role is the correct fix.
WhatBroken StateFixed State
refresh_tokens RLS ENABLED + no INSERT policy → all writes blocked DISABLED (backend-only table, no need for RLS)
login_activity RLS ENABLED + no INSERT policy → audit writes blocked DISABLED (backend-only table)
All other tables RLS enabled with missing service_role policies Explicit service_role ALL USING (true) policy added
JWT_SECRET in .env Missing — token verification could fail silently Added — must match Supabase project JWT secret
auth.js error handling RLS error propagated to login UI as fatal error Non-fatal: warns + continues login even if refresh token store fails

🚨 Step 1 — Run the RLS Fix SQL (DO THIS FIRST)

⚡ CRITICAL — Do This Now

Run rls-fix-v5.1.sql in Supabase SQL Editor

This is the single command that fixes the login error.

  1. Open → Supabase SQL Editor — New Query
  2. Paste the SQL below entirely
  3. Click RUN
  4. You should see: Success. X rows returned. (table list)
  5. Try logging in again — error will be gone ✅
File location: backend/database/rls-fix-v5.1.sql (already created in your project) 
-- ═══ PART 1: DISABLE RLS on backend-only tables ═══
-- These are only accessed via service_role (server-side)
ALTER TABLE public.refresh_tokens    DISABLE ROW LEVEL SECURITY;
ALTER TABLE public.login_activity    DISABLE ROW LEVEL SECURITY;
ALTER TABLE public.audit_logs        DISABLE ROW LEVEL SECURITY;
ALTER TABLE public.collectors        DISABLE ROW LEVEL SECURITY;
ALTER TABLE public.cti_feed_logs     DISABLE ROW LEVEL SECURITY;
ALTER TABLE public.ioc_enrichment_cache DISABLE ROW LEVEL SECURITY;
ALTER TABLE public.threat_feeds      DISABLE ROW LEVEL SECURITY;

-- ═══ PART 2: Drop old incomplete policies ═══
DO $$ BEGIN
  DROP POLICY IF EXISTS "tenant_read_sysmon_sessions"   ON public.sysmon_sessions;
  DROP POLICY IF EXISTS "tenant_read_vulnerabilities"   ON public.vulnerabilities;
  DROP POLICY IF EXISTS "tenant_read_cti_actors"        ON public.cti_actors;
  DROP POLICY IF EXISTS "tenant_read_cti_campaigns"     ON public.cti_campaigns;
EXCEPTION WHEN OTHERS THEN NULL;
END $$;

-- ═══ PART 3: Add service_role bypass for all other tables ═══
ALTER TABLE public.tenants   ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.users     ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.alerts    ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.iocs      ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.cases     ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.playbooks ENABLE ROW LEVEL SECURITY;

CREATE POLICY IF NOT EXISTS "service_role_all_tenants"   ON public.tenants   AS PERMISSIVE FOR ALL TO service_role USING (true) WITH CHECK (true);
CREATE POLICY IF NOT EXISTS "service_role_all_users"     ON public.users     AS PERMISSIVE FOR ALL TO service_role USING (true) WITH CHECK (true);
CREATE POLICY IF NOT EXISTS "service_role_all_alerts"    ON public.alerts    AS PERMISSIVE FOR ALL TO service_role USING (true) WITH CHECK (true);
CREATE POLICY IF NOT EXISTS "service_role_all_iocs"      ON public.iocs      AS PERMISSIVE FOR ALL TO service_role USING (true) WITH CHECK (true);
CREATE POLICY IF NOT EXISTS "service_role_all_cases"     ON public.cases     AS PERMISSIVE FOR ALL TO service_role USING (true) WITH CHECK (true);
CREATE POLICY IF NOT EXISTS "service_role_all_playbooks" ON public.playbooks AS PERMISSIVE FOR ALL TO service_role USING (true) WITH CHECK (true);

-- Verify: show RLS status for all tables
SELECT relname AS table_name, relrowsecurity AS rls_enabled
FROM pg_class
WHERE relnamespace = 'public'::regnamespace AND relkind = 'r'
ORDER BY relname;

🔑 Step 2 — Set JWT_SECRET Correctly

WHERE TO GET IT

Get your exact JWT Secret from Supabase

Go to: Supabase Dashboard → Settings → API → JWT Settings → JWT Secret (Reveal)

The JWT_SECRET you provided (KO3aLoS4LJqK4Nz6O+LxOqq4X/...) has been added to backend/.env. Verify it matches exactly what Supabase shows (including any trailing ==).

Important: In your backend/.env, make sure there are NO quotes around the value and NO extra spaces. The + and / characters are valid base64 and must be preserved exactly. 
# backend/.env — correct format:
JWT_SECRET=KO3aLoS4LJqK4Nz6O+LxOqq4X/MxGlKBbhbiIR6aNg/eyTujTSIuCqMlUT930rBIhIGrorftB0pZjsRux7pR/g==

# ✅ Correct — no quotes
# ❌ Wrong: JWT_SECRET="KO3aLoS4..."  (quotes break dotenv parsing)
# ❌ Wrong: JWT_SECRET = KO3aLoS4...  (spaces break dotenv parsing)

This key is used by backend/config/supabase.js to verify that JWT tokens were genuinely issued by your Supabase project. Without it, token verification can silently fail.
