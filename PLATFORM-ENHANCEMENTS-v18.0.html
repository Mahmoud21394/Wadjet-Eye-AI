⚠️ Platform Settings HTTP 400 — Root Cause & Fix

Root Cause

The original platform-settings.js used PUT /api/settings with the full settings object including null values. The backend validation rejected payloads with null fields (HTTP 400 Bad Request).

Fix Applied in js/settings-enhanced.js

  • Validates all fields before sending (email format, port range, URL validity)
  • Strips all null, undefined, and empty-string values from payload
  • Uses PATCH /api/settings (partial update) instead of full PUT
  • Automatically retries with PUT /api/settings + {settings: {...}} wrapper if PATCH fails
  • Displays specific error messages (validation failed / HTTP error / permission error)
  • Offline edit mode when API is unreachable
// The fix — collect only non-null values and use PATCH:
const { errors, cleaned } = _validateSettings(rawData);
if (errors.length) { show error; return; }
await _settingsEnhFetch('/settings', { method: 'PATCH', body: cleaned });
// cleaned = { platform_name: "X", session_timeout: 480 }
// NOT: { platform_name: "X", smtp_password: null, jira_url: "" }

🔄 Rollback Procedure

Option 1: Git Revert

git revert HEAD --no-edit
git push origin main

Option 2: Render.com Dashboard

Go to Render.com → Your Service → Deploys → click any previous successful deploy → "Manual Deploy"

Option 3: Emergency CSS/JS Disable

If a specific module causes issues, comment out its script tag in index.html and redeploy:

<!-- DISABLED:
<script src="js/darkweb-enhanced.js"></script>
-->
