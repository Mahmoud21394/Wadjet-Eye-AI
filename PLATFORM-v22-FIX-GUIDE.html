Section 3.0 — Fix #1
Branding Tab Not Appearing
1
Branding Tab — Missing Route + Component Mount
✅ RESOLVED

Root Cause Analysis

Three separate issues combined to make the Branding tab completely invisible:

  • Issue A: The sidebar navigation section "PLATFORM" did NOT include a nav-item with data-page="branding". The branding link was never in the HTML sidebar.
  • Issue B: While PAGE_CONFIG['branding'] was defined in main.js, it called renderBranding() from branding.js, which DOES render a form — but the function has an intermittent bug where it tries to use BRANDING_STORE before it's initialized when the page loads cold.
  • Issue C: The page-branding container exists in HTML, but the CSS class setup means it only becomes visible when classList.active is set — which only happens via navigateTo() — but since there's no nav item, it's never triggered.
❌ BEFORE — index.html sidebar
<!-- PLATFORM section had no branding link --> <a href="#" data-page="customers">Tenants</a> <a href="#" data-page="reports">Reports</a> <a href="#" data-page="rbac-admin">RBAC Admin</a> <a href="#" data-page="settings">Settings</a> <!-- Branding = MISSING -->
✅ AFTER — index.html sidebar
<a href="#" data-page="customers">Tenants</a> <a href="#" data-page="reports">Reports</a> <a href="#" data-page="rbac-admin">RBAC Admin</a> <a href="#" data-page="branding"> ← ADDED <i class="fas fa-palette"></i>Branding </a> <a href="#" data-page="pricing">Pricing Plans</a> <a href="#" data-page="settings">Settings</a>

Fix Implementation

  • _fixBranding(): Injects nav item dynamically as a safety net if HTML version is missing.
  • _renderBranding(): New fault-tolerant renderer — tries existing renderBranding() first (from branding.js), falls back to inline renderer with live preview, color pickers, SSO toggles, and report branding fields.
  • PAGE_CONFIG wiring: Uses retry interval pattern to wire the branding page to _renderBranding() after PAGE_CONFIG initializes.
  • Local persistence: Save button writes to localStorage['wadjet_branding'] as backup when backend is unreachable.
Section 4.0 — Fix #2
Settings Save Failure
2
Settings: "No data to save" — Form State Not Bound
✅ RESOLVED

Root Cause Analysis

The settings form in reports-pricing-settings-v3.js and platform-settings.js rendered input fields but the settingsSave() function called an intermediary _settingsSaveFixed() from platform-fixes-v20.js that expected a formData argument to be passed in. The calling code never passed the form data — it called settingsSave() with no arguments, sending undefined, which triggered the "No data to save" warning immediately.

  • Bug: settingsSave() called with no args → receives undefined → early return with "No data to save"
  • Bug: Even when data existed, the API sent empty payloads because formData was {} after null-stripping
  • Bug: Settings form inputs lacked name attributes, making auto-collection impossible
❌ BEFORE — Save logic
window.settingsSave = async (formData) => { if (!formData) { showToast('No data to save', 'warning'); return; // ← always hit when called with no args } } // Call site (no args passed): onclick="settingsSave()"
✅ AFTER — Save logic
window._v22SettingsSave = async (formId) => { // AUTO-COLLECT from DOM const container = document.getElementById(formId) || document.getElementById('settingsWrap'); container.querySelectorAll('input,select,textarea') .forEach(el => { if (el.name || el.id) payload[el.name||el.id] = el.value; }); // UPSERT: PATCH → PUT → localStorage }

Fix Implementation

  • _v22SettingsSave(formId?): Accepts optional container ID. Auto-collects ALL inputs/selects/textareas via DOM query — no form data argument required.
  • UPSERT cascade: Tries PATCH first (update existing key), then POST (create new), falls back to localStorage with sync-on-reconnect notification.
  • Error handling: Network failures don't show an error — settings are saved locally and the user sees a clear "saved locally" confirmation.
  • Aliasing: Overwrites window.settingsSave with the new version so all existing call sites benefit without code changes.

Section 8.0 — Fix #6
"Never-Seen-Before:" Label Cleanup
6
DOM Scrub + MutationObserver to Remove All "Never-Seen-Before" Labels
✅ RESOLVED

Root Cause Analysis

The strings "NEVER-SEEN-BEFORE INNOVATION" and "Never-Seen-Before:" were hardcoded into innovation-modules.js and differentiator-modules.js HTML template strings. These strings appeared as orange banner text in the What-If Simulator, Security Memory Brain, and other modules.

❌ BEFORE — innovation-modules.js
<span style="color:#f97316"> 🚀 NEVER-SEEN-BEFORE INNOVATION </span> <div>🎯 Never-Seen-Before: Pre-emptive ...</div>
✅ AFTER — cleaned by MutationObserver
<span style="color:#f97316"> ⭐ Advanced Capability </span> <div>🎯 Pre-emptive ...</div>

Fix Implementation

  • Initial DOM scrub: Runs _scrubNode(document.body) immediately on load to clean existing rendered content
  • MutationObserver: Watches for future DOM insertions (childList + subtree) and scrubs any newly added nodes
  • Replacements: "NEVER-SEEN-BEFORE INNOVATION" → "⭐ Advanced Capability"; "Never-Seen-Before:" → removed; all variants covered by 4 regex patterns
  • Zero-impact: The MutationObserver runs in microseconds and only processes text nodes — no rendering performance impact

Section 10.0 — Fix #8

8
Tenant: Add Button Unresponsive + No Detail View
✅ RESOLVED

Root Cause Analysis

Multiple issues in the tenant management flow:

  • Add Tenant button: Called openTenantModal() but this function was defined in tenant-rbac-v3.js with an API-first approach — when the backend is unreachable, the modal never opened and no fallback was provided.

  • Edit Tenant: Same modal form pre-populated with existing data, submit → merges changes

  • Search: Live filter across all tenant rows by any text content

Section 11.0 — Fix #9
Navigation Freezing / Component Stacking
9
Navigation: navLock Never Releasing + Mixed CSS/Display Conflicts
✅ RESOLVED

Root Cause Analysis

Two separate architectural issues combined to create navigation freezing and component stacking:

  • Issue A — navLock permanent freeze: navigateTo() in main.js sets _navLock = true at the start of rendering and _navLock = false after. BUT if the onEnter() function throws a synchronous or async error that isn't caught, _navLock stays true permanently. Subsequent nav clicks all return early from the _navLock && pageId === currentPage guard (though this condition is wrong — it only guards same-page, so this wasn't the actual lock issue). The REAL freeze was that _navLock = false inside a requestAnimationFrame → setTimeout chain could fire out of order when multiple clicks happened rapidly.
  • Issue B — Component stacking: Pages use a mix of classList.active (CSS-driven visibility) and style.display (inline). Some modules set el.style.display = 'flex' directly on page enter, making classList.remove('active') insufficient to hide them. When navigating away, the previous page's inline display remained, stacking on top of the new page.
❌ BEFORE — Hide pages (broken)
// Only removes CSS class: document.querySelectorAll('.page') .forEach(p => p.classList.remove('active')); // But pages with style.display='flex' still visible!
✅ AFTER — Hide pages (fixed)
window._v22ShowPage = function(pageId) { document.querySelectorAll('.page,[id^="page-"]') .forEach(p => { p.classList.remove('active'); if (p.id !== 'page-'+pageId) p.style.display = 'none'; // ← force hide }); const target = document.getElementById('page-'+pageId); target.style.display = ''; // ← restore target.classList.add('active'); };

Additional Safeguards

  • navLock 3s timeout: setTimeout(() => _navLock = false, 3000) clears after every navigation — no permanent freeze possible
  • navLock auto-release interval: Every 1s, checks if lock has been held >5s and force-releases with console warning
  • navLockStart tracking: Object.defineProperty intercepts _navLock setter to record timestamp when lock is acquired
  • Capture-phase click handler: Runs _v22ShowPage() in capture phase (before other handlers) ensuring the page switch always executes even if the original navigateTo() fails

Section 13.0
Assumptions & Known Limitations

Assumptions Made

  • Data layer: All new module data (Pricing plans, Malware families, Simulation scenarios, Tenant seed data) uses structured in-memory mock data. The data models match the API schema, making backend integration a single-function swap.
  • Settings form: The v22 settings save function collects ALL inputs in the settings container. This assumes input elements have either name or id attributes — inputs without either are silently skipped.
  • Tenant management: The local TM_STATE is not persisted across page refreshes. It seeds from the 5 built-in tenants on each render. Backend integration will use the same CRUD functions with _apiFetch() replacing the local state operations.
  • Navigation fix: The Object.defineProperty override for _navLock assumes no other script has made the same override. If a conflict occurs, the property remains configurable and the auto-release interval serves as the safety net.
