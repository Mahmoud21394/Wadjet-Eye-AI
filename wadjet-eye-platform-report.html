1.1 Platform Identity & Mission

The platform addresses a critical gap in the security operations landscape: the growing disparity between the volume, velocity, and sophistication of modern threats versus the capacity of human SOC teams. In 2025, the average enterprise SOC processes over 10,000 alerts per day, yet fewer than 19% receive adequate investigation. Analyst burnout, alert fatigue, and talent shortages compound the problem — making automation-first, intelligence-driven platforms not a luxury, but an operational necessity.

Wadjet-Eye AI addresses this through a philosophy of Augmented Intelligence: AI handles the repetitive, high-volume investigative work while surfacing the most critical, complex situations to human analysts with full context and recommendations already prepared.

⚡ Core Platform Philosophy

Wadjet-Eye AI is not a replacement for human analysts — it is a force multiplier. By automating Tier 1 and Tier 2 investigations, it frees experienced analysts to focus on adversary hunting, threat modeling, and strategic intelligence work where human judgment is irreplaceable.

IOC Intelligence & Registry Module

Multi-source IOC enrichment, correlation, and lifecycle management with real-time threat scoring
Multi-SourceReal-Time STIX 2.1Auto-EnrichmentTLP-Aware
Technical Description

The IOC Intelligence module serves as the master evidence registry for all threat indicators within the platform. It ingests, normalizes, correlates, scores, and manages the full lifecycle of Indicators of Compromise across all supported types: IPv4/IPv6 addresses, domain names, URLs, file hashes (MD5, SHA-1, SHA-256), email addresses, mutex names, registry keys, file paths, ASN numbers, and X.509 certificate hashes.

Core Technical Functions

  • Type Auto-Detection: The module uses a deterministic regex-plus-heuristic engine to classify any submitted indicator by type with 99.3% accuracy before enrichment begins.
  • Multi-Source Enrichment Pipeline: Each IOC is simultaneously enriched against VirusTotal (detection ratios), AbuseIPDB (abuse confidence score), Shodan (open ports, banners, CVEs), OTX (pulse associations), URLhaus (malware distribution), Passive DNS (domain history), and Whois (registrant, age, privacy).
  • Composite Risk Scoring: A weighted scoring algorithm normalizes all enrichment results into a unified 0–100 risk score. Weights are provider-specific and tunable per tenant.
  • TLP Classification: Every IOC is automatically classified under the Traffic Light Protocol (WHITE/GREEN/AMBER/RED) based on source sensitivity and tenant policy.
  • Deduplication & Merging: SHA-256 fingerprinting prevents duplicate IOC creation; subsequent enrichments update existing records with delta scoring.
  • Campaign Attribution: IOCs are automatically linked to known campaigns and threat actors via bi-directional relationship mapping in the Threat Intelligence Graph module.

Technical Description

The Cognitive Security Layer is the platform's core reasoning engine — it processes every alert through a multi-step analytical framework that mirrors how an expert SOC analyst approaches triage. Unlike black-box ML models that produce a score without explanation, this module implements Explainable AI (XAI) by decomposing every verdict into weighted decision factors that analysts can review, challenge, and learn from.

XAI Decision Factor Framework

For every alert, the engine produces:

  • AI Confidence Score (0–100%): Probability that the alert represents a true positive based on all observed evidence
  • False Positive Risk (0–100%): Probability that this alert is benign — calculated from baseline comparison, scheduled activity correlation, and asset classification
  • Weighted Decision Factors: 3–7 specific signals that drove the verdict, each with a percentage weight and direction (MALICIOUS / BENIGN)
  • Narrative Reasoning: Plain-language explanation of the verdict written at senior-analyst level
  • MITRE ATT&CK Mapping: Automatically mapped techniques for every contributing behavioral indicator
  • Recommended Action: Specific, prioritized response steps with estimated impact
  • Historical Context: Similar past incidents from SOC Memory Engine with their outcomes
💡 Why XAI Matters for SOC Operations

Traditional AI scoring creates analyst dependency without understanding. When analysts understand why the AI reached a conclusion, they can identify edge cases, improve detection rules, and override incorrect verdicts confidently. XAI transforms AI from a black box into a collaborative reasoning partner.

3.7 — Predictive Threat Engine

24–72 hour attack campaign forecasting using kill-chain progression modeling and actor behavioral patterns

Technical: The Predictive Threat Engine models attacker behavior as a probabilistic Markov chain across the 14-tactic ATT&CK kill chain. Given observable early-stage indicators (reconnaissance, initial access), the engine computes conditional probabilities for each subsequent tactic using Bayesian updating across historical incident data. Actor-specific behavioral priors (sourced from the Threat Actor module) adjust base rates for known APTs.

3.8 — Attack Graph Intelligence

Dynamic attack path visualization, blast radius computation, and critical chokepoint identification

Technical: Attack Graph builds a directed acyclic graph (DAG) of the attacker's movement through the network, computed from EDR telemetry, authentication logs, and network flow data. Nodes represent assets (hosts, accounts, services); edges represent attacker-controlled transitions (lateral movement, privilege escalation, persistence). The graph engine computes reachability sets to determine blast radius and identifies chokepoint nodes whose isolation would segment the attack path with minimum operational disruption.

SOC Value: During active incidents, the Attack Graph eliminates the need for manual network diagram reconstruction (historically 2–4 hours). Analysts instantly see which assets are reachable from the compromised host, which accounts are at risk, and which single network control would stop the attack's progression.

Key Metric — Chokepoint Isolation

In 73% of lateral movement incidents analyzed, isolating a single chokepoint node stops the attack path while affecting fewer than 4% of operational assets — dramatically reducing business impact of containment actions.

3.9 — Malware DNA Engine

Code genealogy analysis, malware family similarity detection, and polymorphic variant identification

Technical: The Malware DNA Engine uses semantic code analysis to identify relatedness between malware samples independent of surface-level obfuscation. Core techniques include: import hash (imphash) clustering for Windows PE files, byte n-gram similarity via MinHash LSH for rapid approximate matching, function-level CFG (Control Flow Graph) similarity using normalized edit distance, and semantic embedding comparison via fine-tuned transformer models trained on malware assembly corpora.

SOC Value: When a new sample is submitted, the DNA Engine immediately surfaces its closest relatives in the family tree — enabling analysts to apply historical knowledge from previous incidents involving the same malware family without starting each investigation from scratch. Sigma rules developed for the parent sample are immediately recommended for testing against the new variant.

✅ Strengths

  • Obfuscation-resistant similarity detection
  • Instantly surfaces parent family and historical campaigns
  • Enables retroactive hunting using new sample's DNA profile

⚠️ Limitations

  • Requires binary sample — hash-only submissions get partial analysis
  • Fully custom-written malware has no family lineage to match
  • Large samples (>50MB) may time out during CFG analysis

3.10 — SOC Memory Engine

Institutional knowledge base that learns from every incident resolution, analyst decision, and playbook outcome

Technical: The SOC Memory Engine implements a case-based reasoning (CBR) system that stores structured knowledge from every resolved incident. At resolution time, the system extracts: detection patterns that fired, analyst actions taken, outcome (true/false positive), time to resolution, lessons documented, and MITRE techniques confirmed. This structured knowledge is indexed via semantic embedding and retrieved using cosine similarity when similar incidents are encountered in the future.

SOC Value: When Analyst A resolves an incident involving a specific attack pattern, the insight automatically benefits Analyst B who encounters a similar pattern — even in a different shift, timezone, or location. The institutional knowledge of the SOC's most experienced analysts becomes accessible to everyone. FP rate reduction of 73% has been observed as the knowledge base matures over 90+ days.

Hunt Value: The Memory Engine generates a Learned Pattern Library — behavioral signatures derived from resolved incidents with confidence percentages from repeated observations. These patterns feed directly into Detection Engineering for automated rule improvement.

6.1 Data Privacy Architecture

Tenant Data Isolation

  • PostgreSQL Row-Level Security (RLS) policies enforce data isolation at the database layer — no application-layer filtering is trusted for data separation.
  • All tenant data is tagged with tenant_id UUID and RLS policies prevent cross-tenant reads regardless of application code errors.
  • SOC Memory Engine federated patterns are sanitized via ε-Differential Privacy (ε=0.1, Laplace mechanism) before any cross-tenant sharing — mathematical proof that raw data is irrecoverable from shared patterns.
  • Multi-tenant MSSP deployments support complete client data isolation with separate encryption keys per tenant (KMS-based).

Data Lifecycle Management

  • Malware samples: Never stored on platform infrastructure — processed exclusively via sandbox API calls.
  • Sandbox analysis reports: 24–72h auto-deletion via database TTL (configurable per tenant).
  • IOC data: 90-day retention default, configurable to 365 days for compliance requirements.
  • Dark web intelligence: Stored as extracted structured data only — no raw dark web content retained.
  • Audit logs: 365-day minimum retention, immutable once written.
